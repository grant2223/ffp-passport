## 1. THE auth.uid() TRAP — affects MEMBERS *and* PROVIDERS
- Members AND providers sign in via a custom JWT (`assets/ffp-api-integration.js` applies it as a global Authorization header). **`auth.uid()` does NOT reliably resolve client-side for them.**
- Any direct `supabase.from(x).insert/update/delete` gated by an RLS policy like `(... = auth.uid())` silently affects **0 rows, no error** → a "Saved/Deleted" toast but nothing changes. Reads work only because live listings are publicly readable.

- **FIX PATTERN:** a SECURITY DEFINER RPC that takes the id explicitly (member id from `FFPAuth.getMember().id`, provider id from `FFP_PROVIDER.id`), `GRANT EXECUTE TO anon, authenticated`.
